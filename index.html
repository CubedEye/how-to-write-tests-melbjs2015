A bit about  

  • Stile is an multi-tenant eLearning platform.
  • Focus around easy of use for teachers to create and manage content for their students.
  • As well as students being able to interact with the same content.
  • We have been developing the product for over 2 years.
  • It's now a big JavaScript application (2.3MB non-gzipped JS, additional 1MB of libraries)
  • With a test suite of ~4000 tests, takes about 5-10+ minutes to run.

A bit about  

So why am I standing here

  • We started with a few flakey tests on our CI server.
  • It wasn't so bad, we reran the CI again and usually passed
  • Over time we had more different flakey tests.
  • Again, it usually all passed on reruns. If first you don't suceed...
  • Occasionally someone would have to "fix" them.
  • Usually the new guy.
  • "Fixing" them was continously running the tests.
  • CI would kindly remind you that it's still not fixed.
  • A few weeks ago our CI server had pass rate of about 30%.
  • We had a serious lack of faith in our test suite.
  • Actual failures were getting lost in the noise.
  • I got the unfortunate job of working out what was going on.

  • A week later...

  • ...I found out that some of our tests sucked!

function flush(model, options) {
  options = options || {};
  var syncOpts = { 'autosaving': true };
  model._$$autosaving = model.sync(model.isNew() ? 'create' : 'update', model, syncOpts)
    .then(null, function(err) {
      if (err instanceof UniqueConstraintViolationError) {
        model.set({'id': err.existingObjectId});
        
        if (model.handleUniqueConstraint) {
          return model.handleUniqueConstraint(err);
        } else {
          return model.sync('update', model, syncOpts);
        }
      } else if (err instanceof syncErrors.VersionConflictError) {
        if (model.handleConflict) {
          return model.handleConflict(err);
        } else {
          stile.core.libs.log.error("Version conflict on model with no handler!", {
              modelAttributes: model.toJSON({all: true}),
              errorMessage: err.message,
              updatedState: err.updatedState,
          });
          return $.Deferred().reject(err);
        }
      } else {
        model._lastAutosaveFailed = true;
        model.trigger('autosaveFail');
        return $.Deferred().reject(err);
      }
    })
    .then(function(res){
      var setOptions = { autosave: true };
      
      model.set({'id': res.id}, setOptions);
      
      if (options.adoptServerAttributes) {
        model.set(res, setOptions);
      } else if (Array.isArray(model.versionNumberKeys)) {
        _.each(model.versionNumberKeys, function(key) {
          if (_.has(res, key)){
            model.set(key, res[key], setOptions);
          }
        });
      }
      model.trigger('autosave', res);
      model._lastAutosaveFailed = false;
    })
    .always(function(){
      model._$$autosaving = null;
    });
    
  return model._$$autosaving;
}

What is a test that doesn't suck?

A test that doesn't suck is Fast

A test that doesn't suck is Fast

  • Not really considered when you start out.
  • Your entire codebase might be 3 or 4 classes and utility functions.
  • The first 10 tests you write are likely to complete in under a second.
  • As the application grows the test suite will have to as well.

  • Ours is currently 4000 or so tests and takes around 5-10 minutes to run.
  • Not unreasonable, it averages 7-16 milliseconds per test.

  • But it's too long to wait between code changes while developing.
  • So tests and code are created in isolation and merged into the code.

A test that doesn't suck is Reliable

A test that doesn't suck is Reliable

  • It should pass consistently. That's it!
  • It should pass on all of your supported platforms.
  • If you can't rely on your tests, why do you have them?
  • Browser JavaScript are hard to keep reliable.
  • Browser quirks, Asynchronous calls (AJAX, Promises), DOM, Events

A test that doesn't suck is Useful

A test that doesn't suck is Useful

  • It should test the behaviour rather than the implementation.
  • It should be resistant to implementation changes. Backbone vs. React
  • It's better to test larger behaviours over smaller units. If it makes sense.

  • Unit test complicated behaviours, like larger asynchronous chains.
  • Integration test simplier common behaviours, like View/Model/Event interaction.
  • Large units of behavioural tests are more robust to refactoring

  • Beware of testing too many small parts, and the underlying frameworks.

A test that doesn't suck is Isolated

A test that doesn't suck is Isolated

  • Was the underlying cause of most of our flakey tests
  • Asynchronous code in scope of the tests was stomping on later tests.
  • Hard to notice early, tests still pass.
  • REST API that were not stubbed, timed out and retried.
  • Holding up the browser because max connections limit was reached.

"Fire and forget. Asynchronous stomping."

  • Something in your code will finish asynchronously after the test completes
  • AJAX times out and tries to update the UI.
  • View has been closed and $() no longer availble, it throws an error.
  • Error bubbles up to window and the test runner fails the currently running test.

Keeping up the quality

  • Make it a code review focus. If the tests look useful and they're passing chances are the code is too.
  • It's hard because the test files are often long and tedious to read.
  • Not a substitute for actually reviewing the code.
// Source file
function add (a, b) { return 4; };

// Test file
describe('test suite', function () {
  it('should do math', function () {
    expect(add(1, 3)).to.equal(4);
  });
});

Keeping up the quality

  • Run a CI environment. Especially if the test runner takes a while to complete.
  • Run the test suite across all supported browsers using tools like Saucelabs.
  • Fail tests if things aren't cleaned up/stubbed correctly. DO THIS EARLY!
  • Behaviour driven development - minimal tests/code for behaviour.
  • Might start with more code, but easier to refactor because testing behaviour.

Keeping up the quality

  • Randomizing the execution order of tests.
  • JSHint your tests - leaky globals, etc.
  • GitHub pull-request hooks with CI.
